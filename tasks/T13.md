## Task: Auto-Locate From IP On Load

**Goal:** A visitor should see their own region's sunset immediately on page load, without typing anything. The request already carries a coarse location (ISP/IP based), so use it. Today everyone lands on an empty centered search.

**Approach - IP geo via Vercel headers, not the browser Geolocation API.** Vercel populates geo headers on every incoming request; reading them costs nothing and never shows a permission prompt. The browser Geolocation API (GPS-accurate, permission dialog) is explicitly out of scope. IP geo is approximate - city-level at best - which is fine for a sunset forecast.

**Header facts (available in any route handler on Vercel):**
- `x-vercel-ip-latitude` / `x-vercel-ip-longitude` - decimal strings.
- `x-vercel-ip-city` - URL-encoded (e.g. `New%20York`) - decode before display.
- `x-vercel-ip-country` - ISO 3166-1 alpha-2, e.g. `US`.
- `x-vercel-ip-country-region` - ISO 3166-2 subdivision code, e.g. `WA` (not the full state name).
- All of them are **absent on local dev** (`npm run dev`) - the feature must degrade silently there.

**Tasks:**

1. **Build `app/api/locate/route.ts`.**
   - Read the headers above from the incoming request. Respond with JSON.
   - Success shape: `{ location: { name, state, lat, lng } }` - the exact `LocationSelection` shape `LocationSearch.onSelect` uses, so the page can feed it through the existing selection path.
   - Respond `{ location: null }` when any of: country is not `US`, lat/lng missing or unparsable, city missing. No error status codes for "not locatable" - it is the normal case, not a failure.
   - Mark the route dynamic (it reads request headers; it must never be statically cached, and the response should be `Cache-Control: no-store` - one visitor's city must not be served to another).

2. **Map the region code to the full state name.**
   - Add `lib/usStates.ts` with a `REGION_TO_STATE` lookup (`"WA" -> "Washington"`, all 50 states + `DC`; territories may map or be dropped - if the code is unknown, still return the location with `state: ""`).
   - This keeps the hero and committed placeholder consistent with what a search selection produces ("Seattle, Washington", not "Seattle, WA").

3. **Wire it into `app/page.tsx`.**
   - On first mount, while still in the empty/centered state, fetch `/api/locate`. If it returns a location, enter the forecast view exactly as if the user had picked that place from the dropdown - same state updates, same committed placeholder ("City, State"), same slow gradient transition. Nothing snaps.
   - If it returns `{ location: null }` or the fetch fails, do nothing - the centered search stays, no error UI, no console noise.
   - **Never fight the user.** If they have started typing or have selected a place before the locate response lands, discard the response. Guard with a ref/flag, and abort the fetch on unmount.
   - The home button keeps working as a reset to the centered search - and returning home must NOT re-trigger auto-locate in that session (locate runs once per page load).

4. **Keep search primacy.** Auto-locate is a starting point, not a lock-in: the search input must remain fully usable from the auto-loaded view (T12 behavior unchanged).

**Acceptance criteria:**
- With `/api/locate` mocked (Playwright route interception) to return Seattle: loading `/` lands directly on the forecast view - hero shows "Seattle, Washington", the input placeholder shows the committed label, gradient renders. No typing needed.
- With `/api/locate` returning `{ location: null }` (which is also the real local-dev behavior): the centered search renders exactly as today.
- With `/api/locate` delayed (mock with latency) and the user typing before it resolves: the locate result is discarded - the dropdown/typed flow wins.
- Home button from an auto-located view returns to the centered search and does not re-locate.
- `curl` the real dev-server `/api/locate` - returns `{ "location": null }` (no Vercel headers locally) with `Cache-Control: no-store`.
- Simulate the deployed path once: `curl /api/locate` with the `x-vercel-ip-*` headers set manually (e.g. Seattle's) - returns the mapped location with the full state name.
- Searching still works from both states (spot-check `seat` -> Seattle per T12).
- No console errors. `npm run build` passes (do not run the build while a dev server is running - stop it first).
- Verify with the `verify` skill (Playwright). Do not add test files.
